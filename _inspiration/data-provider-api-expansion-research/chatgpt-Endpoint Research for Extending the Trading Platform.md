## Provider dossiers

### API Ninjas

API Ninjas is useful when you want **low-friction single-symbol fundamentals and event data** without a complicated envelope. The current URL patterns are simple GET requests with `X-Api-Key` in the header: quote at `/v1/stockprice?ticker=AAPL`, historical bars at `/v1/stockpricehistorical?ticker=AAPL&period=1h&start=<unix>&end=<unix>`, profile at `/v1/ticker?ticker=AAPL`, search at `/v1/tickersearch?name=Apple`, earnings calendar at `/v1/earningscalendar?ticker=MSFT` or `/v1/upcomingearnings?...`, insider activity at `/v1/insidertransactions?ticker=AAPL`, and SEC filings at `/v1/sec?ticker=AAPL&filing=10-K`. The historical endpoint supports `1m`, `5m`, `15m`, `30m`, `1h`, `4h`, and `1d` periods. citeturn11search2turn12search1turn11search0turn13search0turn12search3

The extractor story is straightforward. Quote and profile endpoints return a **single object at the root**; historical, earnings-calendar, insider, and ticker-search endpoints return **root arrays**. The most useful starter mappings are: quote `ticker→symbol`, `price→price`, `updated→timestamp`, `volume→volume`; bars `open→open`, `high→high`, `low→low`, `close→close`, `volume→volume`; profile `name→company_name`, `exchange→exchange`, `website→website`, `cik→cik`, `cusip→cusip`, `isin→isin`; earnings calendar `date→event_date`, `actual_eps→eps_actual`, `estimated_eps→eps_estimate`, `actual_revenue→revenue_actual`, `estimated_revenue→revenue_estimate`; insider `insider_name→owner_name`, `transaction_code→transaction_code`, `shares→shares`, `transaction_value→value_usd`, `filing_date→filing_date`. citeturn11search2turn12search1turn11search0turn13search0

Batching is limited. Quote, profile, and most event endpoints are effectively **per ticker**. Ticker search returns up to **100** results. `/v1/earningscalendar` supports `limit` up to **10** for premium users, while `/v1/upcomingearnings` supports `limit` up to **100** with `offset` pagination. The free quote feed is **15 minutes delayed**; I did not find a current public RPM/day-rate page in the docs I extracted, so treat rate limits as account-plan specific and code for retry plus backoff. citeturn12search1turn11search0turn11search2

### Alpaca

Alpaca is one of the cleanest current fits for your platform if you want a **high-quality batching model**. Market data uses `data.alpaca.markets`, authenticated with `APCA-API-KEY-ID` and `APCA-API-SECRET-KEY`; broker partners use bearer tokens. Current stock quote endpoints are `/v2/stocks/{symbol}/quotes/latest` and `/v2/stocks/quotes/latest?symbols=AAPL,TSLA`. Historical bars are `/v2/stocks/bars?symbols=AAPL,TSLA&timeframe=1Min&start=...&end=...&limit=...`. News is `/v1beta1/news`. Options data uses `/v1beta1/options/quotes/latest`, `/v1beta1/options/snapshots`, `/v1beta1/options/bars`, and `/v1beta1/options/snapshots/{underlying_symbol}`. Dividends and splits ride the single `/v1/corporate-actions` endpoint through the `types=` filter. Profile and ticker-master metadata come from `/v2/assets` and `/v2/assets/{symbol_or_asset_id}` on the trading/paper host, not the market-data host. citeturn15search4turn15search9turn15search2turn16search0turn14search5turn14search0turn14search1turn14search6turn15search0turn16search2turn16search3turn17search0

Your extractor should be designed around Alpaca’s two dominant output forms: **symbol-keyed batch responses** for latest quotes and snapshots, and **paged time-series/event responses** for bars, news, and corporate actions. The bar endpoints are explicitly paginated with `next_page_token`, and the results are sorted by **symbol first, then timestamp**, which is important because a naive extractor can otherwise starve later symbols in a batch when one ticker has a deep history. For news and corporate actions, your canonical flattening keys should include `symbol`, `timestamp/published_at`, `headline/title`, `type`, and the page token. citeturn14search1turn15search2turn16search0

Alpaca’s current rate-limit architecture is public and integration-friendly: the Basic plan allows **200 historical API calls/minute** for equities and options, while Algo Trader Plus advertises **10,000/minute**; docs also explicitly say 429 responses include `X-RateLimit-...` headers. For real-time quote consumption, REST polling is the wrong primitive; use WebSockets wherever possible, and if you must poll on the free/basic tiers, use a **1–5 second cadence** and batch symbols aggressively. citeturn17search0turn17search2turn15search9turn14search5

### Alpha Vantage

Alpha Vantage remains the classic **function-dispatch** provider: one base path, many `function=` values. For your URL builder, the provider-specific logic belongs almost entirely in the function map. Confirmed current functions for your use case are: `GLOBAL_QUOTE`, `TIME_SERIES_*`, `NEWS_SENTIMENT`, `OVERVIEW`, `INCOME_STATEMENT`, `BALANCE_SHEET`, `CASH_FLOW`, `REALTIME_OPTIONS`, `HISTORICAL_OPTIONS`, `DIVIDENDS`, `SPLITS`, `INSIDER_TRANSACTIONS`, `EARNINGS_CALENDAR`, and `SYMBOL_SEARCH`. Representative URLs are `.../query?function=GLOBAL_QUOTE&symbol=IBM&apikey=...`, `.../query?function=TIME_SERIES_DAILY_ADJUSTED&symbol=IBM&apikey=...`, and `.../query?function=SYMBOL_SEARCH&keywords=tesco&apikey=...`. citeturn18search0turn21view1turn21view3turn21view4turn21view5turn21view6turn21view8turn21view0turn21view12

The extractor layer is where Alpha Vantage needs special handling. It is not a uniform JSON API. Quote data sits under `"Global Quote"`. OHLCV sits under a named time-series object such as `"Time Series (Daily)"`. Search results sit under `"bestMatches"`. Financial statements use `"annualReports"` and `"quarterlyReports"`. The newer intelligence and event functions such as news, options, dividends, splits, and insider transactions use separate envelope shapes, so you should model the extractor registry by **function name**, not only by data type. The canonical mapping starter set for daily OHLCV is still the well-known one: `"1. open"→open`, `"2. high"→high`, `"3. low"→low`, `"4. close"→close`, `"5. volume"→volume`. citeturn18search0turn21view1turn21view0

The rate-limit picture has changed enough that old guidance is dangerous. Current official pages clearly state the free tier is **25 requests/day**, while premium plans range from **75 to 1200 requests/minute** and remove daily limits. The support page no longer prominently documents a free-tier per-minute number in the material I extracted, so if your scheduler was built around the historical “5/minute” assumption, update your design notes to treat the publicly confirmed constraint as **25/day** unless your account proves otherwise. Alpha Vantage is therefore not a good provider for quote polling; it is much better as a **low-frequency, high-coverage fundamentals and batch-research source**. citeturn19search1turn20view1

### Benzinga

The current surface is mixed: news is `/api/v2/news`; delayed quotes are `/api/v2/quoteDelayed`; fundamentals are `/api/v3/fundamentals`; earnings/splits/guidance are `/api/v2.1/calendar/...`; dividends are `/api/v2.2/calendar/dividends`; insider SEC-derived data is `/api/v1/sec/insider_transactions/...`; and unusual options activity is `/api/v1/signal/option_activity`. Authentication can still be done with `token=` in the query string, but the docs now recommend the `Authorization: token <API_KEY>` header for production. citeturn22search1turn22search5turn24search1turn22search0turn23search0turn23search1turn24search0turn27search0turn23search4turn28search3

This is one of the easiest providers to extract because the response envelopes are explicit and stable in the examples: news returns a **root array** of articles; fundamentals returns `{ok, data:[...]}`; earnings returns `{earnings:[...]}`; dividends returns `{dividends:[...]}`; splits returns `{splits:[...]}`; insider transactions return `{data:[...]}`; delayed quotes return a **dictionary keyed by symbol**. Pagination is ordinary but generous: page offsets can go to **100000**, news `pageSize` tops out at **100**, and most calendar/insider endpoints allow **1000** records per page. Ticker filters are typically capped at **50** symbols. Public docs confirm 429 throttling exists, but I did not find public per-plan RPM/day numbers in the retrieved docs, so use `updatedSince`/`updated` delta polling on news, calendars, and insider data instead of brute-force time-window rescans. citeturn22search1turn22search0turn23search0turn27search0turn28search4

### EODHD

EODHD is one of the most scheduler-friendly providers because the docs expose **call-unit consumption**, **minute limits**, and **endpoint-specific batch economics**. The strongest current paths are: delayed US quote snapshots at `/api/us-quote-delayed`; end-of-day bars at `/api/eod/{code}`; intraday bars at `/api/intraday/{code}`; news at `/api/news`; fundamentals at `/api/fundamentals/{code}`; dividends at `/api/div/{code}`; splits at `/api/splits/{code}`; insider transactions at `/api/insider-transactions`; and corporate calendars at `/api/calendar/dividends`, `/api/calendar/earnings`, and `/api/calendar/splits`. The provider also explicitly documents search as a separate product and options as a separate US Options dataset. citeturn35search0turn31search2turn32search1turn31search3turn31search0turn34search1turn34search0turn30search1turn34search3turn30search9turn33search0

The extractor model is clean. EOD and intraday OHLCV are **root arrays** of bars. News is a **root array** of articles with standard pagination via `limit` and `offset`. Fundamentals are a **root object containing named sections**, especially `General`, `Highlights`, `Earnings`, and `Financials`; that means your extractor should be “section aware” and able to flatten only the relevant subsection for a requested data type. The best mappings are: EOD bars `date→timestamp`, `open→open`, `high→high`, `low→low`, `close→close`, `adjusted_close→adj_close`, `volume→volume`; fundamentals profile from `General` and metrics from `Highlights`; earnings history and trend from the `Earnings` section; dividends `date→ex_date`, `value→cash_dividend`, `paymentDate→pay_date`, `recordDate→record_date`, `currency→currency`; splits `date→ex_date`, `split→split_ratio` or provider equivalent. citeturn31search2turn32search1turn31search3turn31search0turn34search1turn34search0

The rate-limit and batching story is unusually transparent. Free users are limited to **20 API calls/day** on the free plan example pages; paid subscription plans default to **100,000 call-units/day**; the minute request limit is **1000 requests/minute**; intraday and news cost **5** call-units/request; fundamentals and options cost **10** call-units/request; multi-symbol live requests consume one call-unit per symbol; and rate-limit state is exposed in `X-RateLimit-*` headers. ### Nasdaq Data Link

Nasdaq Data Link should be treated as a **dataset framework**, not as a single stock-data API. The core current API family I verified is the Tables API at `GET /api/v3/datatables/{datatable_code}.{format}?<filters>&api_key=...`. Each dataset product page tells you whether it must be accessed via Tables, real-time/delayed REST, or streaming APIs. The important extractor implication is that **the dataset code is part of the endpoint identity**, not only a query parameter. citeturn43search2turn43search4turn43search3

Pagination and quotas are well documented. Tables are limited to **10,000 rows per call**; the JSON response returns `next_cursor_id`, which you then replay as `qopts.cursor_id`. Authenticated non-premium users get **300 calls per 10 seconds**, **2,000 calls per 10 minutes**, and **50,000 calls/day**; premium tables subscribers get **5,000 calls per 10 minutes** and **720,000 calls/day**. Anonymous access is much tighter. The Tables API also returns normal HTTP error codes and custom 429-style error codes such as `QELx01`. citeturn43search0turn43search6turn43search9

For your platform, that means Nasdaq Data Link is best used for **institutional or premium datasets** such as fundamentals, holdings, and alternative data where you already know the target table code. It is a weak fit for a generic `quote/ohlcv/news` abstraction unless you curate specific datatables ahead of time and register them explicitly in your provider config. citeturn43search2turn43search4turn43search7

History should be normalized into a canonical bar record with `date`, `open`, `high`, `low`, `close`, and `volume`; the docs note that the endpoint can also price options if you pass an OCC symbol. citeturn42search7turn42search9turn42search1

Tradier supports both single-call multi-symbol quotes and a POST quote path for larger lists, which makes it attractive for scheduled quote batches. I did not verify current public rate-limit numbers in the retrieved docs, so I would treat real-time polling as account-plan specific and rely on streaming for anything above low-Hz usage. citeturn42search2turn42search6turn42search4

## Builder and extractor design

The right abstraction is a **provider capability registry**, not just a provider URL registry. At minimum, each provider entry should declare: `builder_mode` (`simple_get`, `function_get`, `dataset_get`, `post_body`), `auth_mode` (`header`, `query`, `bearer`), `multi_symbol_style` (`csv`, `repeated_filter`, `body_array`, `none`), `pagination_style` (`offset_limit`, `next_page_token`, `next_cursor_id`, `none`), and `extractor_shape` (`root_object`, `root_array`, `wrapper_array`, `symbol_keyed_dict`, `named_section_object`). That single schema is enough to encode the biggest differences between Alpaca, Alpha Vantage, Benzinga, EODHD, FMP, Nasdaq Data Link, and Tradier. citeturn17search0turn18search0turn22search1turn36view0turn37search3turn43search0turn42search10

In practice, your response extractor registry should be keyed by **provider + endpoint family**, not only provider + data_type. Alpha Vantage alone proves why: `GLOBAL_QUOTE`, `TIME_SERIES_DAILY_ADJUSTED`, `NEWS_SENTIMENT`, `OVERVIEW`, and `SYMBOL_SEARCH` all return different envelopes. EODHD fundamentals have sectioned JSON, Benzinga quote/news/calendar routes use three different shapes, Alpaca switches between symbol-keyed dictionaries and paginated arrays, and Tradier wraps search results under nested keys. If you collapse those into one extractor per provider, you will create a brittle parser surface. citeturn18search0turn21view1turn21view0turn22search1turn24search1turn31search0turn15search2turn16search0turn42search7

For field mapping, start with a **canonical core** that is deliberately small: `symbol`, `timestamp`, `open`, `high`, `low`, `close`, `adj_close`, `volume`, `bid`, `ask`, `headline`, `published_at`, `company_name`, `exchange`, `industry`, `sector`, `eps_actual`, `eps_estimate`, `revenue_actual`, `revenue_estimate`, `cash_dividend`, `split_ratio`, `owner_name`, `filing_date`, `source_url`, `currency`, `cik`, `cusip`, `isin`. Every provider you wire should first map into that shared core and only then expose extended provider-native fields as `extras`. That lets your cron pipelines and downstream portfolio analysis remain provider-agnostic even when the upstream shapes differ sharply. citeturn11search2turn22search1turn38search2turn42search9

## Recommended rollout

The endpoints that should be wired **first** for a retail portfolio-analysis tool are: quotes, daily/intraday OHLCV, company profile, fundamentals, earnings calendar/history, dividends, splits, and news. Across your list, the highest-confidence near-term implementation stack is: **Alpaca or Tradier for quotes and batches**, **EODHD or FMP for historical prices, profile, and broad fundamentals**, **Benzinga or Alpaca for news**, **Alpha Vantage/EODHD/Benzinga for corporate actions and calendars**, and **FMP 13F plus Benzinga/API Ninjas insider data for ownership-style workflows**. citeturn17search0turn42search2turn31search2turn38search2turn22search1turn16search9turn23search0turn23search1turn39search0turn13search0
