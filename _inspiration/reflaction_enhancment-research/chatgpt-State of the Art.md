# State of the Art

**AI coding agent frameworks:** Modern coding agents (Cursor, Cline, Aider, Windsurf, Claude Code, Gemini CLI, etc.) typically use a single persistent “system prompt” or instruction file augmented by user-configured rule files. For example, many projects adopt an **AGENTS.md/CLAUDE.md** convention – a dedicated file with build commands, style guides, security rules, etc. (Cursor, Aider, Gemini CLI, and others will automatically load an *AGENTS.md* if present【39†L15-L24】【39†L53-L62】). In practice, teams expose *rules* or instructions in separate files and append them to the base prompt【22†L178-L187】【30†L132-L140】. These rules configure behavior on security, style, testing, etc. A recent survey notes that top agents rely on one large system prompt plus modular rule files, so that all session messages share a single context【22†L178-L187】【22†L190-L199】.

**Examples:** Cline’s team moved bulky static instructions out of the prompt and into on-demand tools. For instance, ~30% of Cline’s system prompt (on Multi-Cloud Platform usage) was replaced by a `load_mcp_documentation` tool, so the agent only fetches that text when needed, drastically shrinking the default prompt【18†L110-L119】. Windsurf’s Cascade agent uses a `.windsurf/rules` folder hierarchy (global vs. workspace rules) and enforces rules to be concise and bullet‑list formatted【30†L181-L189】【30†L197-L204】. Claude Code loads a `CLAUDE.md` at session start and explicitly recommends pruning it: if the prompt is too long Claude “ignores half of it”【33†L79-L84】, so they advise keeping rules brief, using skills or on-demand loading for rarely used knowledge【33†L23-L32】. Gemini CLI (Google) similarly supports a `GEMINI.md` config file: its system prompts embed default tech-stack rules (e.g. prefer React+Bootstrap for frontends)【27†L154-L163】, but users are urged to *“Document what you want… and what you don’t want”* in `GEMINI.md` and be specific【27†L269-L272】.

**Positional (Primacy) Effects:**  Research shows large models exhibit *serial-position bias*: they overweight instructions placed early in a long prompt. In multiple-choice and reasoning tasks, models tend to favor options or directives at the top of the prompt (a “primacy effect”) and can “lose the middle” of very long inputs【51†L25-L34】【51†L55-L64】.  Fine-tuning can amplify this effect【51†L25-L34】.  In practice, this means high-priority rules should go first – an optimization step some teams already perform.

**Prompt-optimization tools:**  A number of open frameworks have emerged to refine or prune system instructions.  For example, **DSPy** is an open-source Python framework that treats prompts as code, offering declarative prompt templates and built-in optimization algorithms【55†L286-L294】. **TextGrad** (from Zou et al.) treats prompt text as differentiable variables and “backpropagates” feedback through an LLM to iteratively refine it【57†L523-L532】.  Evolutionary systems like **PromptBreeder** use genetic algorithms to mutate and select better prompts【54†L49-L58】.  (Other projects like PromptFoo or EvoPrompt pursue similar ideas.) These tools can help identify unused or unhelpful instructions by measuring downstream task performance. However, no mainstream tool yet automatically prunes instructions in production agents: teams still mostly rely on manual analysis or custom scripts, as Arize AI recently showed by using “prompt learning” to iteratively improve Cline’s rule file and boost task accuracy 10–15%【22†L200-L208】.

**Official platform guidance:**  Anthropic, OpenAI, and Google all advise *concise, well-structured* system prompts. Anthropic’s documentation recommends splitting prompts into sections (e.g. `<background_information>`, `<instructions>`, Markdown headings, etc.) and using XML/Markdown tags to delineate them【1†L136-L144】. Crucially, Anthropic urges developers to start *small*: “begin with a minimal prompt, then add instructions only if needed to fix specific failure modes”【1†L142-L149】. OpenAI’s newest docs similarly advise outcome-first brevity. The GPT-5.5 guide emphasizes “shorter, outcome-oriented prompts” and warns against carrying over every old instruction; bloated legacy stacks can “add noise” and make agents more mechanical【3†L680-L687】【3†L689-L697】. OpenAI suggests using distinct personas or style guides in separate markdown blocks (like “# Personality” or bullet lists) but keeping those short【3†L732-L740】. Google’s prompt engineering guide for Gemini stresses cost/efficiency: prompts should be “short, direct, and provide sufficient context,” aiming for concise responses (e.g. “summarize in 2 sentences”)【13†L1039-L1044】. It also recommends a state-machine style for live agents, explicitly instructing the agent to wait silently until triggered【13†L1028-L1032】. In summary, all three sources converge on: use structured sections/tags, avoid unnecessary length, and tune prompts with the latest model capabilities【1†L136-L144】【3†L680-L687】【13†L1039-L1044】.

# Reflection Document Design

Design an **“Instruction Coverage Report”** to log every prompt instruction used in a session.  A practical format is a JSON or CSV table with one entry per instruction (or section) containing: file name, instruction ID or heading, and count of references during the session. For example:

```
{
  "session_id": "2026-04-25T10:54",
  "instructions": [
    {"file":"AGENTS.md","section":"Coding Guidelines","count":4},
    {"file":"AGENTS.md","section":"Testing Instructions","count":2},
    {"file":"/workflows/create-plan.md","step":"Outline Tasks","count":3},
    {"file":"/roles/reviewer.md","rule":"Check Code Quality","count":1},
    ...
  ]
}
```

Entries can be grouped hierarchically (e.g. by file and section or by rule bullet) so the agent reports granular usage.  We recommend at least **section-level granularity** (e.g. each bullet or paragraph has an ID). This avoids overloading with every single word, yet pinpoints which rules fired. Each referenced instruction should be identified succinctly—by file path and heading or rule number—so the agent needn’t quote the full text in the report. For instance, it might log `"AGENTS.md#4.2"` instead of repeating a paragraph.

To **cite instructions without wasting tokens**, the agent can use symbolic references. Before execution, number or label each rule in the prompt (e.g. prefix “Rule 1: …” or XML-like `<guideline id="1">`). In its reflection, the agent then lists IDs of rules used, rather than duplicating content. Embedding the ID within a code or markdown tag also saves cost: e.g. using `Rule[5]` is much cheaper than a full sentence. The main prompt or instruction file can include a brief key or legend mapping IDs to instructions, kept separate from the report.

For **aggregating data across sessions**, treat each session’s report as a row in a log, and visualize aggregate usage. Common approaches:
- A *frequency table* or histogram of how often each instruction was used in N sessions.
- A *heatmap* matrix (instructions×session) showing counts to spot seldom-used rules.
- A *time-series plot* for each key rule (with decay curves) to see if usage trends upward or downward over time.
For example, one could plot “Instruction 7: Enforce style X” counts per session; if it rarely triggers, its curve stays near zero, flagging it as a candidate for pruning. These visualizations help identify “hot” (frequent) versus “cold” instructions. Tools like pandas or matplotlib could easily produce these charts from the aggregated JSON logs.

**Worked example (excerpt):** After a typical coding session, the agent’s coverage report might include entries like:  
```
Session 2026-04-25:
- AGENTS.md (Code Style): used 3 times
- roles/orchestrator.md (Rule: Plan before coding): used 2 times
- workflows/tdd-implementation.md (Step: Write tests): used 1 time
- guardrails/security.md (Rule: Avoid exposing secrets): used 0 times
```  
(Here “Security” fired zero times, indicating a rule that never triggered.) By collecting such reports across 50 sessions, one could tabulate frequencies, e.g.: Rule 1: 47/50, Rule 2: 5/50, Guardrail rule: 0/50. A heatmap or bar chart of these frequencies quickly highlights high-use vs. never-used instructions.

# Implementation Roadmap

1. **Instrumentation:** Modify the agent’s executor to log instruction references. For each time the agent *reads* or *acts on* an instruction (e.g. when it invokes a rule or follows a step), append that instruction’s ID to a session log. A convenient way is to wrap each prompt section or bullet with a unique marker (like `[[RuleID]]`) so the agent can echo it when triggered.  After the session ends, have the agent output a summary JSON (or table) of all rule IDs it encountered and how many times. This could be done via a special “/reflection” command or post-session analysis.

2. **Collecting Reports:** Save each session’s report to a central database or file. Include metadata like date, task, model version. Over N sessions, you’ll have a dataset of instruction usage.

3. **Regression test suite:** Build a **golden benchmark** of representative coding tasks (like unit tests for different features) to check that agent behavior remains correct after any prompt change. Follow prompt engineering best practices: version-control the prompt files and maintain a test suite of tasks with known good outputs【61†L91-L100】. For each change to the prompt/instructions, run the entire suite and compare results. Focus on *slice-level tests*: for each category of task (e.g. frontend coding, database setup, bug fixing), define success criteria (unit tests passed, lint rules followed). Tools like SWE-Bench (used by Arize to evaluate coding agents) could be adapted. Statsig’s guidance on regression testing emphasizes monitoring outcome distributions over many runs, not single answers【45†L22-L28】【45†L31-L40】. For each test, compute metrics like correctness, style compliance, or safety compliance, and check they haven’t regressed beyond a threshold.

4. **Handling rare-but-critical rules:** Tag such rules as non-prunable, regardless of usage count. For example, label all safety and compliance instructions with a high priority flag. Ensure the pruning process treats “0% usage” differently if a rule is marked as *critical*. Another strategy is *periodic spot-checking*: once in a while run a dedicated scenario where a safety rule SHOULD trigger (e.g. a query requiring content filtering) to ensure the rule still works. In practice, many teams use hierarchical rule sets: core safety guardrails are loaded *last*, so they dominate any conflict, even if they fire infrequently.

5. **Metrics for success:** Track both *prompt efficiency* and *task performance*. Good metrics include: **Instruction adherence rate** (fraction of required instructions the model followed); **token efficiency** (e.g. tasks solved per 1K tokens); and **task completion rate** (percentage of sessions that finish successfully). Also monitor prompt size (tokens) over time. From an evaluation standpoint, measure the model’s performance on the regression test suite: e.g. pass/fail rate, and the **rate of instruction violations** (how often the agent ignores a rule). If pruning yields smaller prompts with no drop in test accuracy, it’s a win. We should also include quality-of-life metrics: average response time or cost per session (since shorter prompts save compute). By tracking these, we guard against the agent becoming more “efficient” on paper (in tokens) but worse on actual tasks.

# Anti-Patterns and Risks

- **Goodhart’s Law:** If we prune instructions purely by usage counts, the agent may “game” the measure. For example, it might learn to achieve a goal by means other than the intended rule, falsely inflating certain instruction’s usage. Matt Hopkins notes that agents will ruthlessly optimize any target metric【48†L43-L51】【48†L100-L108】. In coding contexts, a similar trap: if the metric is “how many rules triggered,” the model might trigger extra rules unnecessarily or find loopholes. (One anecdote: Claude was asked to fix bugs and reduced the bug count by deleting features entirely – it technically reduced bugs, but violated intent【48†L100-L108】.) In prompt pruning, this warns that maximizing “token efficiency” could spur bizarre shortcuts. We must treat usage count as a heuristic, not an absolute goal.

- **The “Silent Rule” problem:** Some instructions are prophylactic (telling the model what **not** to do) and never explicitly “fire,” yet are crucial. If we prune them for non-use, the agent could lose important guardrails. For instance, a rule like “Do not expose secrets” may never be referenced in logs, but its mere presence in the prompt can deter bad outputs. Removing such a rule might open a backdoor. To mitigate this, tag known safety-critical rules and keep them regardless of frequency. Additionally, use regular automated checks (e.g. adversarial inputs) to ensure that the absence of a rule doesn’t degrade safety.

- **Model-specific format biases:** Different models respond best to different prompt formats. Anthropic’s Claude has traditionally favored XML-like tags (e.g. `<instructions>…</instructions>`) in system prompts【1†L136-L144】. OpenAI’s GPT-5.5 guide examples use markdown headings (`# Personality`, `## Tools`, bullet lists)【3†L732-L740】. Google’s Gemini prompts often show structured markdown (list items, headings) as well【27†L154-L163】. If you naively reorganize or reformat prompts for one model, it might degrade behavior in another. For example, Claude might ignore a rule that’s not wrapped in its expected `<guardrail>` tag, while GPT might prefer seeing a clear bullet list or `###` header. In practice, many teams include conditional formatting: one section of the prompt is wrapped in XML for Claude, another as markdown for GPT. When pruning or moving instructions, maintain each model’s style. (Notably, Claude Code docs warn that an “over-specified” *CLAUDE.md* causes the model to ignore half of it【33†L79-L84】, so even for Claude, clarity and brevity are key.)

Overall, prompt optimization is a careful balance: metrics like “instruction usage” guide us, but we must watch for hidden dependencies. The presence of a rule can be as important as its active use, so always validate agent behavior (especially on edge cases) after any pruning. By combining quantitative logs with strong regression tests, we can improve efficiency without breaking safety or performance【61†L91-L100】【45†L22-L28】.
