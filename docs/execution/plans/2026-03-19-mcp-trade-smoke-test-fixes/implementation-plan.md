# MCP Trade Tool Smoke Test Fixes

Fix the `create_trade` MCP tool confirmation token schema gap discovered during live smoke testing. The tool's Zod `inputSchema` is missing the `confirmation_token` parameter, preventing agents from completing the destructive-action confirmation flow on static/annotation-unaware clients.

## Root Cause Analysis

### Bug 1: `create_trade` — confirmation_token not in inputSchema ❌ REAL BUG

| Component | Detail |
|-----------|--------|
| **File** | [trade-tools.ts](file:///p:/zorivest/mcp-server/src/tools/trade-tools.ts#L26-L64) |
| **Root cause** | `inputSchema` defines 9 fields but omits `confirmation_token`. The MCP SDK Zod schema strips unknown params before they reach the handler. |
| **Middleware** | [confirmation.ts:137](file:///p:/zorivest/mcp-server/src/middleware/confirmation.ts#L137) checks `params.confirmation_token` on static clients |
| **Spec** | [05-mcp-server.md:889](file:///p:/zorivest/docs/build-plan/05-mcp-server.md) — "destructive tools require a `confirmation_token`" |
| **Impact** | Agents cannot create trades on annotation-unaware clients (Cursor, etc.) |
| **Fix** | Add `confirmation_token: z.string().optional()` to the `inputSchema` |

### Bug 2: `update_settings` — 422 response ✅ NOT A BUG (false positive)

The 422 during smoke testing was caused by testing with `{"test_key": "test_value"}` — an unregistered setting key. The API's `SettingsService.bulk_upsert()` validates keys against the settings registry and correctly rejects unknown keys. The MCP tool sends the correct body format (`params.settings` serializes to a flat dict like `{"theme": "dark"}`). The existing test at [settings-tools.test.ts:97-128](file:///p:/zorivest/mcp-server/tests/settings-tools.test.ts#L97-L128) confirms correct behavior.

## Feature Intent Contract

### AC-1: Schema accepts `confirmation_token`
`create_trade` inputSchema includes `confirmation_token` as an optional string parameter. *Source: Spec*

### AC-2: Token flows through to middleware
When an agent calls `create_trade` with `confirmation_token`, the middleware receives and validates it before reaching the handler. *Source: Spec*

### AC-3: Token NOT forwarded to API
The `confirmation_token` field is consumed by the middleware and NOT included in the REST body sent to `POST /api/v1/trades`. *Source: Local Canon*

### AC-4: Backward compatibility
Calling `create_trade` without `confirmation_token` on dynamic clients still passes through (no regression). *Source: Spec*
