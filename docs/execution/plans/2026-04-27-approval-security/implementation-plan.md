# Implementation Plan: Approval Security & Validation Hardening

> **Project**: `2026-04-27-approval-security`
> **Build Plan Section(s)**: [09g](../../../build-plan/09g-approval-security.md) §1–§2, [09f](../../../build-plan/09f-policy-emulator.md) (extension)
> **Status**: `draft`

---

## Goal

Harden the Zorivest scheduling pipeline against agent self-approval and close MCP tool gaps that force agents to bypass the MCP confirmation layer.

**Three MEUs in dependency order:**

1. **MEU-PH11** — CSRF approval token architecture. Electron IPC generates single-use, policy-scoped, time-limited tokens that the `POST /policies/{id}/approve` endpoint validates. The renderer's `approvePolicy()` function injects the token in the `X-Approval-Token` header. AI agents cannot forge tokens because they don't have access to Electron's main process.

2. **MEU-PH12** — MCP scheduling gap fill. Add `delete_policy` (destructive + confirmation), `update_policy` (in-place update), and `get_email_config` (SMTP readiness check) MCP tools. New `GET /settings/email/status` backend endpoint returns minimal readiness info without exposing SMTP credentials. Prevents agents from bypassing MCP guards by calling REST directly.

3. **MEU-PH13** — Emulator VALIDATE phase hardening. Add EXPLAIN SQL pre-check (catches schema errors before execution), SMTP config validation for send steps, and step output wiring validation (render→send chain integrity).

**Security rationale:** Without PH11, an agent can self-approve policies via REST. Without PH12, agents must call REST directly to delete/update policies, bypassing MCP's confirmation gate. Without PH13, the emulator misses infrastructure-level errors that only surface at runtime.

## User Review Required

> [!IMPORTANT]
> **IPC Bridge Design Decision (PH11):** The spec (09g §9G.1e) offers two approaches for the FastAPI server to validate approval tokens generated by Electron:
>
> 1. **HTTP callback** — Electron main exposes `http://127.0.0.1:{port}/internal/validate-token`. FastAPI calls this endpoint during approval. **Recommended: simpler, no filesystem races, single TCP roundtrip.**
> 2. **File-based token exchange** — Electron writes tokens to a temp file with restrictive permissions. FastAPI reads and validates. Cons: filesystem race conditions, permission management on Windows.
>
> **This plan proposes HTTP callback (Option 1).** The spec requires an ADR documenting the choice. The ADR will be created as a subtask.
>
> **Alternative considered but rejected: HMAC shared secret** — Electron passes a signing key to FastAPI at startup via env var. Token = HMAC(secret + policy_id + timestamp). Rejected because the shared secret would be readable from the process environment, which an agent with shell access could exploit.

### MEU-PH11: CSRF Approval Token

#### Boundary Inventory

| Surface | Schema Owner | Field Constraints | Extra-Field Policy |
|---------|-------------|-------------------|-------------------|
| `POST /policies/{id}/approve` header `X-Approval-Token` | `validate_approval_token` FastAPI dependency | 64-char hex string, 5-min TTL | N/A (header) |
| IPC `generate-approval-token` | Electron main handler | `policyId: string` (UUID) | Ignored — positional arg |
| IPC `validate-approval-token` (internal HTTP) | Electron HTTP callback | `token: string`, `policyId: string` | `extra="forbid"` on JSON body |

#### Acceptance Criteria

| AC | Description | Source | Negative Test |
|----|-------------|--------|---------------|
| AC-1 | `generate-approval-token` IPC returns `{token: string, expiresAt: number}` with 64-char hex token | Spec: 09g §9G.1c | Non-string policyId rejected |
| AC-2 | Token is single-use: second validation with same token fails | Spec: 09g §9G.1d | Replay returns `TOKEN_NOT_FOUND` |
| AC-3 | Token expires after 5 minutes (TTL = 300,000ms) | Spec: 09g §9G.1d | Expired token returns `TOKEN_EXPIRED` |
| AC-4 | Token is policy-scoped: token for policy A rejected for policy B | Spec: 09g §9G.1d | Cross-policy returns `POLICY_MISMATCH` |
| AC-5 | `POST /policies/{id}/approve` without `X-Approval-Token` returns 403 | Spec: 09g §9G.1c | Missing header → 403 with descriptive message |
| AC-6 | `POST /policies/{id}/approve` with invalid/random token returns 403 | Spec: 09g §9G.1c | Random string → 403 |
| AC-7 | `POST /policies/{id}/approve` with valid token succeeds (200) | Spec: 09g §9G.1c | N/A (happy path) |
| AC-8 | Periodic cleanup removes expired tokens from memory (60s interval) | Spec: 09g §9G.1c | No memory leak after TTL |
| AC-9 | Internal validation HTTP callback bound to 127.0.0.1 only | Local Canon: security boundary | External IPs cannot reach endpoint |
| AC-10 | Renderer `approvePolicy()` calls `generate-approval-token` IPC and injects `X-Approval-Token` header on `POST /approve` | Spec: 09g L24–35 (flow diagram) | Approve without IPC call fails with 403 |
| AC-11 | Renderer approval flow test verifies IPC→header injection chain | Spec: 09g L24–35 | Mock IPC returning null → approve call fails |
| AC-12 | ADR documenting IPC bridge design choice exists | Spec: 09g §9G.1e | N/A (document) |
