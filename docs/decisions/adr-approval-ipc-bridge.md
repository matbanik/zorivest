# ADR: Approval Token IPC Bridge — HTTP Callback

## Context

The Zorivest scheduling pipeline requires CSRF-style approval tokens to prevent AI agents from self-approving policies via the REST API. Tokens are generated by Electron's main process (via IPC from the renderer) and validated by the FastAPI server on `POST /policies/{id}/approve`.

The FastAPI server runs as a child process of Electron. It needs a mechanism to validate tokens against the in-memory store in the Electron main process.

## Decision

**Use an internal HTTP callback.** Electron main starts a minimal HTTP server bound to `127.0.0.1` on a dynamic port. When FastAPI receives an approval request with an `X-Approval-Token` header, it calls `POST http://127.0.0.1:{port}/internal/validate-token` with the token and policy ID. Electron validates against its in-memory store and returns the result.

## Alternatives Considered

### File-based token exchange

Electron writes tokens to a temp file with restrictive permissions. FastAPI reads and validates from the file.

**Rejected because:**
- Filesystem race conditions between write and read
- Windows ACL management adds complexity
- File cleanup required on crash/restart
- No advantage over HTTP on localhost

### HMAC shared secret

Electron generates a signing key and passes it to FastAPI via environment variable at startup. Token = HMAC(secret + policy_id + timestamp).

**Rejected because:**
- The shared secret is readable from the process environment
- An agent with shell access could extract the key and forge tokens
- Violates the "process-bound" security property

## Consequences

- **Positive:** Single TCP roundtrip, no filesystem races, token store remains in-memory (process-bound), simple implementation
- **Positive:** Port is dynamic (assigned by OS), reducing collision risk
- **Negative:** Requires the Electron main process to be running for approval to work (acceptable — the GUI is the approval authority)
- **Negative:** If the internal HTTP server crashes, approvals are blocked until restart (fail-closed, which is the desired behavior)
