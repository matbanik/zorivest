# MEU-2 Handoff: StrEnum Definitions

## Scope

Implement the 14 `StrEnum` classes defined in `docs/build-plan/01-domain-layer.md` §1.2 with comprehensive TDD coverage. Pure enum definitions — no infrastructure, no cross-module imports.

Build plan reference: [01-domain-layer.md §1.2](../../docs/build-plan/01-domain-layer.md)

## Feature Intent Contract

### Intent Statement

The domain exposes the exact `StrEnum` definitions listed in `01-domain-layer.md` §1.2, with exact class names and string values, and no extra enum classes.

### Negative Cases

- Must NOT use `Enum`, `IntEnum`, or plain string constants instead of `StrEnum`
- Must NOT "correct" enum values using other docs when they conflict with `01-domain-layer.md` §1.2
- Must NOT broaden scope beyond `enums.py`, `test_enums.py`, and minimal import wiring
- Must NOT invent a 15th enum class not present in the build plan

## Design Decisions & Known Risks

- **Decision**: No `__init__.py` re-exports — **Reasoning**: Tests import directly from `zorivest_core.domain.enums`; re-exports would be scope creep and could conflict with MEU-3 — **ADR**: inline (minor)
- **Decision**: `DisplayModeFlag` docstring kept — **Reasoning**: Only enum with a docstring in the build plan — preserved verbatim — **ADR**: inline (minor)
- **Decision**: 14 enums, not 15 — **Reasoning**: Build plan §1.2 explicitly defines 14 classes. MEU registry description corrected from "15" to "14" in this session — **ADR**: inline (corrective)
- **Assumption**: Build plan §1.2 member values are the golden reference for all string assertions
- **Risk**: No `__init__.py` re-exports means downstream MEUs import from `zorivest_core.domain.enums` directly
