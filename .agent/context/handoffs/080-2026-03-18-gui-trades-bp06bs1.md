# MEU-47: GUI Trades Page

## Scope

Full trades page with TanStack Table (9 columns), trade detail panel with Info/Journal/Screenshots tabs, screenshot management, and trade journal. Implements build plan [06b §Trades & Journal](../../../docs/build-plan/06b-gui-trades.md).

## Feature Intent Contract

### Intent Statement
Users can view, sort, paginate, and select trades in a data table. Selecting a trade opens a detail panel with form editing, journal entry, and screenshot management.

### Acceptance Criteria

- AC-1 (Source: Spec): `TradesTable` renders 9 columns: Time, Instrument, Action, Qty, Price, Account, Comm, P&L, 📷
- AC-2 (Source: Spec): Table supports column sorting (click header) and pagination (50 rows/page)
- AC-3 (Source: Spec): Row selection opens `TradeDetailPanel` in 60/40 split layout
- AC-4 (Source: Spec): Detail panel has 3 tabs: Info, Journal, Screenshots
- AC-5 (Source: Spec): Info tab has 9 form fields (instrument, action, qty, price, account, commission, realized_pnl, notes) with Zod validation
- AC-6 (Source: Spec): Journal tab has star ratings (setup/execution), followed-plan select, emotional state dropdown, lessons textarea, tag chips
- AC-7 (Source: Spec): Screenshot panel has file upload, Ctrl+V paste, thumbnail grid, lightbox viewer
- AC-8 (Source: Spec): Action column color-codes BOT=green, SLD=red; P&L positive=green, negative=red
- AC-9 (Source: Spec): All TRADES.* data-testid attributes from test-ids.ts are applied

### Negative Cases

- Must NOT: render trade detail panel until a row is selected
- Must NOT: allow saving with empty instrument or non-positive quantity/price

## Design Decisions & Known Risks

- **Decision**: Client-side pagination (narrowed contract) — **Reasoning**: The UI fetches the first 50 trades via `?limit=50&offset=0` and paginates that slice in-memory using TanStack Table's `getPaginationRowModel()`. The backend route already supports `limit`/`offset` for future cursor-based server-side pagination. Upgrading to server-driven pages is deferred until trade volumes exceed the first-page fetch — no API changes needed. **Source basis**: pragmatic MVP scope; backend contract is forward-compatible.
- **Decision**: Zod validation via `zodResolver` — **Reasoning**: Consistent with project's validation approach. Schema reuse possible for API-side validation.
- **Decision**: Screenshot panel uses `onPaste` handler — **Reasoning**: Works in Electron's renderer process. Full Electron clipboard API integration deferred.
